# 实施计划

> 本文档记录项目 v1.0.0 内部测试版的历史实施路线，并在后半部分补充后续功能优化的工程执行顺序。产品级修改说明保留在 `memory-bank/modification-plan.md`，本文档用于指导后续按 Step 开发、验收和更新进度。

## 阶段一：项目初始化（Step 1-3，已完成）

### Step 1: 创建 Next.js 项目骨架
**任务**：
- 使用 create-next-app 初始化项目
- 配置 TypeScript + Tailwind CSS
- 安装 shadcn/ui 基础组件：Button、Input、Card、Dialog、Select、Form、Dropdown Menu、Toast、Sheet

**验证方式**：
- 本地运行 `npm run dev`，浏览器访问 http://localhost:3000 能看到默认页面
- 检查页面无报错

---

### Step 2: Firebase 项目配置
**任务**：
- 登录 Firebase 控制台，创建项目
- 开启 Authentication（邮箱密码登录）
- 开启 Firestore 数据库
- 获取项目配置信息（apiKey 等）

**验证方式**：
- 通过开发的 App 注册测试用户
- 注册后能在 Firestore 看到用户数据

---

### Step 3: Firebase 集成
**任务**：
- 在项目中安装 firebase SDK
- 创建 firebase.ts 初始化文件
- 封装 Auth 和 Firestore 工具函数

**验证方式**：
- 登录页面能正常调用 Firebase 登录/注册 API
- 新用户数据能写入 Firestore

---

## 阶段四：复盘系统（Step 11-13，部分实现）

### Step 11: 日复盘
**任务**：
- 在饮食记录界面添加复盘入口
- 自动生成日复盘内容
- 显示：今日执行度、触发性进食次数、做得好的1件事、明日优先改进1件事
- 以上全部由系统分析当日数据自动生成

**验证方式**：
- 一天的饮食记录完成后能看到日复盘
- 复盘内容根据记录数据自动生成

---

### Step 12: 周复盘
**任务**：
- 饮食记录界面包含日复盘和周复盘两个 Tab
- 自动生成周复盘内容
- 显示：体重周均变化、围度变化、饮食结构概览
- 分析：情绪-进食关联、高风险场景TOP3
- 输出：下周3条可执行策略（系统自动推荐）
- 后续版本：根据用户历史数据（加餐原因、情绪等）综合分析

**验证方式**：
- 一周的记录完成后能在饮食记录页看到周复盘
- 高风险场景分析逻辑正确
- 系统能自动推荐可执行策略

**当前说明**：
- 独立 `/dashboard/review` 页面尚未实现，已列入 `modification-plan.md` 的第一阶段。

### Step 13: 复盘联动计划

### Step 16: 菜谱生成（增强版）
**任务**：
- 添加条件约束引擎

- 时间约束：每餐可投入时长
- 根据用户历史习惯（易饿时段、偏好做法）调整菜单
- 用户使用一周后，学习历史数据（加餐原因、情绪等）综合推荐

**验证方式**：
- 设置条件后生成的菜单符合约束

## 阶段七：核心问题修复（Step 20-23，待执行）

### Step 20: 仪表盘体重同步与文案调整
**任务目标**：
- 修复已录入今日体重后，仪表盘“今日体重”未同步显示的问题。
- 统一“初始体重、今日/最新体重、目标体重”的产品口径。
- 让进展概览基于最新体重动态计算剩余差距。

**修改范围**：
- 仪表盘今日体重改为读取当天体重记录子集合。
- 今日体重优先取晨起体重；若无晨起体重，则取当天最新体重。
- 将目标设定中的“当前体重”展示文案调整为“初始体重”。
- 进展概览使用今日体重或最近一次体重与目标体重计算。
- 今天没有体重记录时，使用最近一次体重作为参考，并在界面标注。

**验证方式**：
- 在体重记录页新增今日体重后，回到仪表盘能立即看到。
- 仪表盘显示“今日体重、初始体重、目标体重”三项。
- 进展概览的“还需”随最新体重变化。

**影响文档**：
- 完成后更新 `memory-bank/progress.md`。
- 若调整体重数据读取口径，更新 `memory-bank/architecture.md`。

### Step 23: 运动记录支持自定义运动量单位
**任务目标**：

**验证方式**：

### Step 28: 菜谱生成规则和库存不足提示
**任务目标**：
- 让菜谱生成遵循更明确的搭配规则。
- 避免生成用户没有录入的食材。
- 在库存不足时给出明确提示和补货方向。

**修改范围**：
- 生成菜谱时只使用用户库存中的食材。
- 建立基础搭配规则：
  - 早餐：主食 + 蛋白/蛋奶，可选水果。
  - 午餐：2 份蔬菜 + 1 份蛋白质 + 1 份主食。
  - 晚餐：蔬菜 + 蛋白质，主食可选。
- 库存不足时，不强行生成完整一周菜单。
- 输出补货清单，例如：蛋白质不足、蔬菜不足、主食不足。

**验证方式**：
- 菜谱不会默认出现用户未录入的食材。
- 库存不足时，页面明确提示“当前库存不足以生成完整一周菜谱”。
- 页面能给出需要补充的食材类别。

**影响文档**：
- 完成后更新 `memory-bank/progress.md`。
- 更新 `memory-bank/architecture.md` 中菜谱生成规则、库存约束和补货提示逻辑。
- 必要时同步更新 `memory-bank/modification-plan.md` 的任务状态。
