# 好食家功能修改计划

## 目标

本计划用于记录好食家后续功能修改的优先级、拆分方式和验收标准，便于逐步 vibecoding，也便于后续其他协作者理解当前进度。

当前修改重点分为三类：

1. 修复已影响使用的 bug 和缺失页面。
2. 优化移动端记录体验和数据管理能力。
3. 调整信息架构，将强相关功能合并，并升级菜谱生成体验。

2026-04-26 新增一轮迭代重点：

1. 允许未登录用户完整只读浏览，降低首次访问门槛。
2. 强化真实记录原则，明确历史记录不支持修改。
3. 优化个人资料、体重、围度、饮食、库存和仪表盘的信息结构。
4. 将界面视觉从黑白后台感升级为偏冷色、时尚、个性的冰蓝玻璃感。
5. 后续引入身体基础资料和轻量营养目标，但不做医疗级饮食处方。

2026-04-27 新增视觉调和重点：

1. 当前纸白米色方格背景、黑色侧边栏和 lime 牛油果绿高亮之间气质不统一。
2. 下一轮视觉改版从“高对比游戏化”调整为“轻食手账 + 轻量任务板”。
3. 保留记录任务、完成度和连续记录反馈，但去掉大面积黑色，降低高饱和 lime 的刺激感。
4. 通过更轻盈的食物色、暖白卡片和可爱但克制的小图标，让界面更贴近减脂记录、菜谱计划和个人手账场景。

2026-04-27 新增记录路径与食材保质期重点：

1. 侧边栏应优先呈现高频记录和做饭相关入口，降低从首页进入核心任务的路径成本。
2. “食材与菜谱、饮食记录、运动记录”应前置到“首页”之后。
3. 添加食材时不再填写抽象的“剩余天数”，改为填写真实到期日期，例如 `2026-04-30`。
4. 系统基于真实日期判断食材是否已过期、今日到期或临近到期，减少用户手动估算。

2026-04-27 新增今日记录删除与移动端空间优化重点：

1. 体重记录、饮食记录、运动记录应补齐与围度记录一致的今日记录删除能力，用于处理当天误录。
2. 历史记录继续遵循“真实记录用于复盘”的原则：今天之前的数据不支持修改，也不扩展任意历史编辑入口。
3. 移动端应减少统计卡片和任务卡片的纵向堆叠，优先将短摘要卡片调整为 2 列网格。
4. 仪表盘、饮食记录、运动记录、体重记录、复盘页等存在小卡片单列堆叠的位置，应统一检查并做适度压缩。

## 阶段一：先修明显问题，恢复核心闭环

优先级最高。此阶段目标是让现有功能变得可信、可用，不做大规模页面重构。

### 1. 仪表盘体重同步与文案调整 ✅

问题：
- 已录入今日体重后，仪表盘“今日体重”没有同步显示。
- “当前体重”更准确的含义是目标设定时的起始体重，容易和今日体重混淆。
- “进展概览”没有基于最新体重动态更新。

修改方向：
- 仪表盘今日体重改为读取当天体重记录子集合。
- 今日体重优先取晨起体重；若没有晨起体重，则取当天最新体重。
- “当前体重”更名为“初始体重”。
- 进展概览使用“今日体重/最新体重”和目标体重计算剩余差距。
- 如果今天没有记录体重，则使用最近一次体重作为参考，并在界面上标注清楚。

验收标准：
- 在体重记录页新增今日体重后，回到仪表盘能立即看到。
- 仪表盘显示“今日体重、初始体重、目标体重”三项。
- 进展概览的“还需”随最新体重变化。

完成状态：
- 已完成，完成日期：2026-04-25。
- 用户验证测试通过。
- 保留内部字段 `currentWeight`，界面按“初始体重”展示。

### 2. 新建独立复盘页面 ✅

问题：
- 导航和首页快速入口指向 `/dashboard/review`，但该页面不存在，导致 404。
- 复盘是产品核心功能，不适合只藏在饮食记录页里。

修改方向：
- 新建 `/dashboard/review` 页面。
- 将日复盘、周复盘能力迁移或复用到独立复盘页。
- 饮食记录页可保留轻量复盘入口，但完整复盘以独立页面为主。

验收标准：
- 点击侧边栏“复盘”不再 404。
- 复盘页面能展示今日复盘和本周复盘。
- 无记录时展示明确的空状态和引导入口。

完成状态：
- 已完成，完成日期：2026-04-25。
- 用户验证测试通过。
- 已新增 `/dashboard/review` 独立复盘页，完整复盘以独立页面为主。
- 饮食记录页保留原有轻量复盘入口，并与独立复盘页复用同一套分析逻辑。

### 3. 围度记录移动端优化与删除能力

问题：
- 手机端围度数据卡片纵向占用过长。
- 已录入的围度历史数据不支持删除。

修改方向：
- 移动端围度摘要卡片改为两列显示。
- 卡片文案和字号压缩，减少纵向高度。
- 为围度记录增加删除能力。
- 优先支持删除今日记录；如需要删除任意历史记录，再扩展历史列表的数据结构。

验收标准：
- 手机端围度摘要不再单列过长。
- 今日围度记录列表中可以删除误录数据。
- 删除后今日记录和趋势图同步刷新。

### 4. 运动记录支持自定义运动量单位 ✅

问题：
- 当前运动记录固定使用“时长/分钟”，无法表达步数、公里数、次数等。

修改方向：
- 将运动记录从单一 `duration` 扩展为“运动量数值 + 单位”。
- 单位支持：分钟、步、公里、次、组、圈、自定义。
- 卡路里仍保持可选。
- 运动记录列表显示用户选择的单位。
- 今日统计从“运动时长”调整为更通用的“运动量记录”，或按单位分组汇总。

验收标准：
- 用户可以记录“走路 8000 步”。
- 用户仍可以记录“跑步 30 分钟”。
- 旧数据兼容显示为“分钟”。

完成状态：
- 已完成，完成日期：2026-04-25。
- 用户验证测试通过。
- 保留旧 `duration` 兼容逻辑，新记录使用“运动量数值 + 单位”展示和汇总。

## 阶段二：合并强相关页面，降低使用路径成本

此阶段目标是减少导航割裂，让用户按任务流操作。

### 5. 合并目标设定和体重记录 ✅

问题：
- 目标设定和体重记录本质都属于体重管理，目前拆成两个入口，使用路径割裂。

修改方向：
- 将“目标设定”整合进“体重记录”页面。
- 页面结构建议：
  - 顶部：目标摘要，显示初始体重、目标体重、今日/最新体重、剩余差距。
  - 中部：快速记录今日体重。
  - 下部：趋势图、历史记录、目标设置表单。
- 侧边栏移除独立“目标设定”入口，或改为跳转到体重页面的目标区域。

验收标准：
- 一个页面内可以完成目标设置和体重记录。
- 仪表盘、体重页对“初始体重/今日体重/目标体重”的口径一致。

完成状态：
- 已完成，完成日期：2026-04-25。
- 用户验证测试通过。
- 目标设置已合并进体重记录页，侧边栏不再展示独立目标设定入口。
- 旧目标设定路由保留为兼容跳转，避免已有入口造成 404。

### 6. 合并食材库存和菜谱生成 ✅

问题：
- 菜谱生成强依赖食材库存，当前拆成两个页面，用户需要来回切换。

修改方向：
- 新页面可命名为“食材与菜谱”。
- 页面内使用 Tab 或分区：
  - 食材库存
  - 一周菜谱
  - 条件设置
- 侧边栏减少为一个入口。

验收标准：
- 用户在同一页面添加食材后，可以直接生成菜谱。
- 菜谱生成能即时读取最新库存。

完成状态：
- 已完成，完成日期：2026-04-25。
- 用户验证测试通过。
- 食材库存与菜谱生成已合并为“食材与菜谱”页面，侧边栏保留一个入口。
- 旧菜谱生成路由保留为兼容跳转，避免已有入口造成 404。

## 阶段三：升级菜谱生成和导出能力

此阶段改动最大，建议单独实施和验证。

### 7. 菜谱展示压缩为表格/日历视图 ✅

问题：
- 当前菜谱生成后篇幅过长，不方便一眼查看一周安排。

修改方向：
- 桌面端使用一周表格：日期、早餐、午餐、晚餐。
- 移动端使用紧凑日期卡片或横向表格。
- 支持视图切换：
  - 表格视图
  - 九宫格视图
  - 日历视图

验收标准：
- 一周三餐可以在较短页面内完整查看。
- 手机端不需要大量滚动才能看完主要内容。

完成状态：
- 已完成，完成日期：2026-04-25。
- 一周菜单支持表格、九宫格和日历三种视图切换。
- 桌面端表格压缩展示完整一周三餐，移动端使用紧凑日期卡片。
- 视图切换共用同一份菜谱数据，内容保持一致。

### 8. 菜谱导出为便于查看的一页纸 ✅

问题：
- 用户希望将菜谱导出后便于日常查看。

修改方向：
- 优先实现浏览器打印/保存 PDF。
- 提供专用打印样式。
- 打印模板支持：
  - 一页纸表格
  - 九宫格
  - 日历形式

验收标准：
- 用户点击导出后，可以通过浏览器保存为 PDF。
- 打印内容不包含无关按钮、导航和表单。

完成状态：
- 已完成，完成日期：2026-04-25。
- 一周菜单区域新增“导出 / 保存 PDF”入口，直接复用浏览器打印/保存 PDF。
- 打印内容仅包含当前菜单视图，支持表格、九宫格和日历三种菜单形式。
- 打印时会隐藏导航、按钮和表单，并通过白名单方式仅保留菜谱主内容。

### 9. 菜谱生成遵循更明确的搭配规则 ✅

问题：
- 当前三餐搭配不够合理。
- 午餐没有严格遵循类似 211 的搭配原则。
- 生成结果可能包含用户没有录入的食材。

修改方向：
- 生成菜谱时只使用用户库存中的食材。
- 建立基础搭配规则：
  - 早餐：主食 + 蛋白/蛋奶，可选水果。
  - 午餐：2 份蔬菜 + 1 份蛋白质 + 1 份主食。
  - 晚餐：蔬菜 + 蛋白质，主食可选。
- 库存不足时，不强行生成完整一周菜单。
- 输出补货清单，例如：蛋白质不足、蔬菜不足、主食不足。

验收标准：
- 菜谱不会默认出现用户未录入的食材。
- 库存不足时，页面明确提示“当前库存不足以生成完整一周菜谱”。
- 页面能给出需要补充的食材类别。

完成状态：
- 已完成，完成日期：2026-04-26。
- 菜谱生成已改为只使用用户库存中的食材，不再使用内置推荐食材兜底。
- 早餐、午餐、晚餐按基础搭配规则生成，其中肉类和蛋奶作为蛋白质来源。
- 库存不足时不生成完整一周菜单，并提示主食、蔬菜、蛋白质等补货方向。
- 水果作为早餐可选补充，缺失时只显示建议，不阻断生成。

## 阶段四：入口体验、视觉风格与记录可信度

此阶段目标是让新用户能先看懂产品，再在需要保存数据时登录；同时建立更有辨识度的视觉基调，并强化“真实记录用于复盘”的产品原则。

### 10. 未登录用户完整只读浏览 ✅

问题：
- 当前新用户打开网页后只能看到登录/注册入口，无法先了解应用内部结构和核心功能。
- 招聘作品集或开源访问场景下，访问者可能不愿意一开始就注册。

修改方向：
- 未登录用户可以访问仪表盘和各功能页面。
- 未登录状态下页面展示只读内容、空状态、功能说明或示例状态。
- 所有写入动作需要登录或注册，包括记录、保存、删除、生成个人计划、编辑资料等。
- 未登录点击写入动作时，弹出登录/注册提醒，而不是直接跳转或静默失败。

验收标准：
- 未登录访问 `/`、`/dashboard` 和各 dashboard 子页面不会被强制跳转到登录页。
- 未登录点击任一保存/记录/删除动作时，看到明确的登录/注册提醒。
- 未登录状态不会向 Firestore 写入数据。
- 已登录用户原有记录流程不受影响。

当前状态：
- 已在 Step 29 落地，详见 `memory-bank/progress.md`。

### 11. 冰蓝玻璃感视觉升级 ✅

问题：
- 当前界面以黑白灰为主，默认后台模板感较强，缺少作品集展示时的视觉辨识度。

修改方向：
- 整体视觉改为偏冷色、时尚、个性的冰蓝玻璃感。
- 使用冷白、浅蓝灰、蓝青色作为主基调。
- 通过半透明卡片、细边框、柔和阴影建立层次。
- 保持记录工具的信息密度和可读性，不做营销页式大 Hero。
- 优先统一 layout、侧边栏、顶部栏、Card、Button、Tabs、Input，再逐步处理页面局部配色。

验收标准：
- 页面不再呈现单一黑白灰后台感。
- 主要页面在移动端和桌面端都保持清晰可读。
- 视觉变化不影响表单、记录列表、图表和打印样式的可用性。

当前状态：
- 已在 Step 30 落地，详见 `memory-bank/progress.md`。

### 12. 真实记录原则提示 ✅

问题：
- 用户希望体重、饮食、运动等记录保持真实，不鼓励事后修改历史数据。
- 当前页面没有明确说明历史记录不可修改的产品原则。

修改方向：
- 首页和各记录页增加统一提示：记录用于复盘，请尽量真实填写；历史记录暂不支持修改。
- 今日误录可以保留现有删除或重记能力。
- 不扩展任意历史记录编辑能力。

验收标准：
- 首页、体重、围度、饮食、运动等记录入口能看到真实记录提示。
- 历史记录不可修改的原则在界面上表达清楚。

当前状态：
- 已在 Step 31 落地，详见 `memory-bank/progress.md`。

## 阶段五：个人资料与记录体验优化

此阶段目标是优化高频记录页面的信息结构，让记录、查看和复盘更符合真实使用场景。

### 13. 昵称与个人资料 ✅

问题：
- 当前右上角只能显示邮箱账号，不适合作为日常使用称呼。
- 后续身体指标和营养目标需要基础资料支持。

修改方向：
- 支持用户设置昵称。
- 顶部栏和仪表盘欢迎语优先显示昵称，没有昵称时回退邮箱。
- 用户资料预留身高、性别、出生年份或年龄、活动水平等字段，后续用于 BMI 和营养目标。

验收标准：
- 用户可以保存昵称。
- 顶部栏和仪表盘欢迎语显示昵称。
- 未设置昵称时继续显示邮箱，不影响旧用户。

当前状态：
- 已在 Step 32 落地，详见 `memory-bank/progress.md`。
- 同步新增个人资料页入口，右上角昵称和头像区域可点击进入编辑。

### 14. 体重目标顶部折叠与 7 天趋势修正 ✅

问题：
- 体重目标设置放在页面底部，和体重记录任务流关系弱。
- 当前 7 天趋势只显示有数据的日期，容易出现只显示昨天和今天的情况。

修改方向：
- 体重目标摘要放到页面顶部。
- 通过“设置目标”按钮展开目标表单，默认折叠。
- 7 天趋势固定显示今天及过去 6 天的日期刻度。
- 缺失日期不画点、不错误连线，只连接有数据的趋势。

验收标准：
- 用户进入体重页即可看到目标摘要和设置目标入口。
- 点击“设置目标”后可以展开并保存目标。
- 7 天趋势始终显示 7 个日期刻度。
- 缺失数据不会被当作 0 或错误连线。

当前状态：
- 已在 Step 33 落地，详见 `memory-bank/progress.md`。

### 15. 围度最近有效记录展示 ✅

问题：
- 围度不一定每天记录，如果今天没记录但昨天记录了，摘要显示空会造成误解。
- 当前部分空字段可能以 0 参与展示，不符合真实语义。

修改方向：
- 围度顶部 4 个摘要显示最近一次有效围度记录。
- 摘要中明确标注记录日期，例如“最近记录：2026-04-25”。
- 某一项没有有效记录时，该项显示“未记录”，不使用 0。

验收标准：
- 今天没有围度记录但历史有记录时，摘要显示最近记录值和日期。
- 单项缺失时不会显示 0 cm。

当前状态：

问题：
- “剩余天数”需要用户自己估算，不够直观，也容易随着时间推移失真。
- 食材是否过期应由系统根据真实日期判断，而不是依赖用户录入一个不会自动变化的数字。

修改方向：

- 食材数据新增或改用 `expiryDate` 字段，格式建议为 `YYYY-MM-DD`。
- 库存列表基于当前日期和 `expiryDate` 自动计算状态：
  - 已过期
  - 今日到期
  - 临近到期
  - 未到期
- 旧数据如果仍只有 `remainingDays`、没有 `expiryDate`，库存列表应提示“需要补充到期日期”。
- 未补充 `expiryDate` 的旧食材不参与菜谱生成，避免系统把保质期不确定的食材安排进菜单。

验收标准：

- 日期临近时，库存列表给出临近到期提醒。
- 旧食材数据仍能正常展示，不因为字段变化导致页面报错。
- 旧食材缺少 `expiryDate` 时明确提示需要补充到期日期。
- 旧食材缺少 `expiryDate` 时，不会被菜谱生成使用。

当前状态：

- 本次未做旧数据批量迁移，也不按 `remainingDays` 自动推算真实到期日期。

## 阶段十一：今日记录删除与移动端空间优化

此阶段目标是补齐高频记录页的误录处理能力，同时让手机端信息排列更紧凑。原则是只处理当天误录，不开放历史记录编辑，避免破坏复盘数据的真实性。

### 27. 体重、饮食、运动补齐今日记录删除

问题：
- 围度记录已经支持删除今日误录，但体重、饮食、运动页面的今日记录还缺少同等能力。
- 用户当天误录后只能继续新增记录，容易让今日摘要和复盘结果被错误数据干扰。
- 项目已经明确历史记录不支持修改，因此不能扩展成任意历史编辑。

修改方向：
- 复用现有 `deleteDailyRecord`，只对 `today` 对应日期下的记录执行删除。
- 体重记录页的今日记录列表增加删除按钮和确认弹窗。
- 饮食记录页的今日饮食记录按餐次展示时，每条记录增加删除按钮和确认弹窗。
